# 이 책을 읽는 방법

이 mdBook은 `cpal` API 레퍼런스를 그대로 옮긴 문서가 아니다.

목표는 다음 두 가지다.

- `Phase 1 - mod_player` 아키텍처를 스스로 설계할 수 있게 만들기
- 예제를 읽을 때 "왜 이렇게 써야 하는가"를 설명할 수 있게 만들기

## 추천 독서 순서

1. 이 장에서 전체 흐름을 먼저 잡는다.
2. `cpal이 해결하는 문제`와 `Host -> Device -> Stream` 구조를 이해한다.
3. 오디오 콜백과 실시간 스레드 제약을 먼저 머리에 넣는다.
4. 그 다음에 440Hz 사인파 예제를 읽는다.
5. 마지막으로 DSP 체인 삽입 구조와 디버깅 체크리스트를 본다.

## 이 책에서 예제를 읽는 방법

예제는 완성 코드를 복붙하라고 넣은 것이 아니다.

예제를 읽을 때는 항상 아래 순서로 본다.

1. 이 예제가 해결하는 문제가 무엇인지 확인
2. 어떤 상태가 콜백 밖에 있어야 하는지 확인
3. 어떤 값이 오디오 스레드에서만 사용되는지 확인
4. 실시간 오디오 규칙 위반이 없는지 확인
5. 이 구조가 `RuStudio`의 `mod_player`에 어떻게 연결되는지 확인

## 학습 완료 기준

이 책을 다 읽고 나면 적어도 아래 질문에는 바로 답할 수 있어야 한다.

- `cpal`이 하는 일과 하지 않는 일은 무엇인가?
- 왜 `build_output_stream()` 콜백이 오디오 엔진의 핵심 진입점인가?
- 왜 `Mutex::lock()`이나 동적 할당을 콜백 안에서 피해야 하는가?
- 440Hz 사인파 예제가 왜 `mod_player`의 최소 성공 기준인가?
- `Vec<Box<dyn AudioModule>>` 같은 DSP 체인을 어느 계층에서 소유해야 하는가?

이 질문에 자연스럽게 답할 수 있으면, 이제 구현 전에 아키텍처 스케치를 시작해도 된다.
