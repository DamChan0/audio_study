# Chapter 1 - 이 책을 읽는 방법

이 책은 DSP를 **수학 공식 모음**으로 읽으면 안 된다.

수식이 아니라 "이 한 샘플을 만들 때 어떤 계산이 일어났는가"라는 관점으로 읽는다.

## 모든 DSP 블록을 동일한 3개 질문으로 본다

이 책에서 등장하는 모든 처리 블록은 다음 3개 질문으로 분해된다.

```text
1. 입력 (input)        : 이 블록은 무엇을 받는가?
2. 출력 (output)       : 이 블록은 무엇을 내보내는가?
3. 내부 상태 (state)   : 이 블록은 직전 샘플의 무엇을 기억하는가?
```

이 세 가지를 한 블록당 종이에 직접 적어보면, 어떤 DSP든 같은 골격으로 보이기 시작한다.

예를 들어 보자.

```text
oscillator
  input  : (없음)
  output : f32 샘플 한 개
  state  : 현재 위상(phase)

gain
  input  : f32 샘플 한 개
  output : f32 샘플 한 개
  state  : (없음, 현재 gain 값은 파라미터지 상태가 아님)

delay
  input  : f32 샘플 한 개
  output : f32 샘플 한 개
  state  : 과거 N개 샘플을 담은 버퍼 + 쓰기 인덱스
```

이 책의 모든 챕터는 이 세 줄 표를 채울 수 있게 만드는 것이 목표다.

## 읽으면서 계속 의식할 것

- "이 코드 한 줄이 1초에 몇 번 실행되는가?" → 보통 48,000번 또는 96,000번이다.
- "이 변수는 콜백 안에서 매 샘플 새로 만들어지는가, 아니면 콜백 밖에 살아 있는가?" → 매 샘플 만들면 거의 다 stateless, 살아 있으면 stateful.
- "이 처리는 진짜 mod_player 콜백에 넣어도 안전한가?" → 할당/lock/IO 없는가?

## 학습 완료 기준

이 책을 다 읽고 나면 아래 질문에 막힘없이 답할 수 있어야 한다.

- DSP는 cpal과 무엇이 다른가?
- 샘플과 프레임은 같은 말인가?
- sine oscillator의 "상태"는 무엇이고, 왜 그것을 기억해야 하는가?
- `linear *= 0.5` 와 `db -= 6.0`은 같은 결과인가?
- envelope의 "현재 단계(stage)"는 왜 enum으로 표현하기 좋은가?
- delay buffer가 ring buffer와 무엇이 다른가, 무엇이 같은가?

답이 막히는 항목이 있으면 그 챕터로 돌아간다.
